When state logic becomes complex (many ways to update the same state), useState inside context becomes hard to manage. This is where we introduce the "Action-based" pattern using useReducer.

1Ô∏è‚É£ Create reducer

AuthContext.js 
import React, { createContext, useReducer } from "react";

export const AuthContext = createContext();

const initialState = {
  user: {
    name: "John",
    role: "Admin",
    email: "john@example.com"
  }
};

function authReducer(state, action) {
  switch (action.type) {
    case "LOGIN":
      return { ...state, user: action.payload };

    case "LOGOUT":
      return { ...state, user: null };

    case "UPDATE_EMAIL":
      return {
        ...state,
        user: { ...state.user, email: action.payload }
      };

    default:
      return state;
  }
}

export function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, initialState);

  return (
    <AuthContext.Provider value={{ state, dispatch }}>
      {children}
    </AuthContext.Provider>
  );
}

2Ô∏è‚É£ Use It

ProfileSettings.js 
import { useContext } from "react";
import { AuthContext } from "../AuthContext";

function ProfileSettings() {
  const { dispatch } = useContext(AuthContext);

  return (
    <button onClick={() => dispatch({ type: "LOGOUT" })}>
      Logout
    </button>
  );
}

export default ProfileSettings;

üéØ What Students Learn

useReducer gives us:

  • Centralized logic: All business rules for updating state are in one function (the reducer).
  • Predictable updates: You can see exactly what happens for every "Action".
  • Action-based system: Components just say *what* happened, not *how* to change the state.
  • Easier to scale: Adding new features just means adding a new case to the switch statement.

Now say to students:

"If our app grows even bigger... what if we have Auth, Cart, Theme, Notifications, Products, and Orders? Do we create 6 separate contexts?"

This leads us to the ultimate solution: Redux.
